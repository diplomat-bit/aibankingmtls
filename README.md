## 3. Architectural Framework: "Iron and Dirt"
The `aibanking.dev` platform operates on three foundational pillars, designed to be the operating system for the AI-driven economy.

### 3.1 Sovereign Nodes (`/server.ts`)
The central nervous system of the platform. It is not just an Express server; it is a sovereign node that orchestrates the flow of data between compute events and financial settlement. It integrates Vite as middleware for development while ensuring a robust, production-ready environment for financial orchestration, acting as the secure gateway that bridges the gap between raw inference and real-world liquidity.

### 3.2 The Cryptographic Heart (`/src/services/ledgerService.ts`)
The engine of the "UUID mining logic." It is the most critical component of the architecture, where every financial event is cryptographically linked to a compute event. By using `firebase-admin` to persist these events immutably to Firestore, it ensures that the ledger is not just a database, but a verifiable, audit-proof record of truth that bridges the gap between the digital and financial realms.

### 3.3 The Command Console (`/src/components/LedgerDashboard.tsx`)
The human-machine interface for the new economy. It provides real-time visualization of the "mining" process, allowing operators to monitor the conversion of compute-level events into financial entries. It is designed for high-density information, providing immediate, immutable feedback on the state of the ledger, ensuring that the human operator is always in sync with the machine's financial output.

### 3.4 The Arteries of Liquidity (`/src/services/plaidService.ts` & `/src/services/blockchainService.ts`)
These services are the vital arteries of the system. `plaidService.ts` connects the sovereign ledger to the traditional banking system, providing the necessary liquidity for real-world settlement. `blockchainService.ts` acts as the bridge to the decentralized economy, enabling cross-chain asset movement and ensuring that the Sovereign Compute Ledger is not just a silo, but a truly interoperable financial engine.

## 4. Technical Specification

### 4.1 API Specification
The platform exposes a RESTful API to manage the ledger and payment flows.

| Endpoint | Method | Description |
| :--- | :--- | :--- |
| `/api/ledger/entry` | `POST` | **Core:** Mines a new ledger entry linking a compute event to a financial event. |
| `/api/modern_treasury/accounts` | `GET` | Retrieves internal account balances. |
| `/api/modern_treasury/counterparties` | `GET` | Lists counterparties for payment orchestration. |
| `/api/plaid/create-link-token` | `POST` | Initiates Plaid link for bank connectivity. |

### 4.2 Data Flow Mapping: Compute to Settlement
1. **Compute Event:** GPU inference generates a result.
2. **UUID Mining:** `ledgerService.createLedgerEntry(computeEventId, ...)` is called.
3. **Ledger Entry:** A record is created linking the compute UUID to a financial amount in Firestore.
4. **Settlement:** The system triggers a payment order via Modern Treasury using the ledger entry as the source of truth.

## 5. Security and Compliance
The Sovereign Compute Ledger is built on a "Default Deny" security model.

*   **mTLS/FAPI Compliance:** All nodes communicate over mTLS, ensuring that only authorized sovereign nodes can participate in the ledger network.
*   **Firestore Rules:** We implement strict, prototype-level Security Rules that enforce least-privilege access, data validation, and immutability.
*   **Identity Integrity:** Every financial event is cryptographically bound to a verified compute event UUID, preventing spoofing and ensuring the integrity of the ledger.
